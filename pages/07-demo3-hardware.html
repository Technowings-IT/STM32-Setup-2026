Demo 3 — Screen ↔ hardware

Goal: tap an on-screen button to toggle the on-board LED, show the physical user button's state on the screen, and log both over UART. This is the full round-trip — TouchGFX for the UI, CubeMX for the pins, CubeIDE for the behaviour — and the complete Model-View-Presenter pattern.

What you'll learn

  • Going back to ⚙️ CubeMX to enable GPIO + UART on a TouchGFX project
  • Driving hardware from the UI, and pushing hardware state to the UI
  • The full Model · Presenter · View wiring, and where each file lives
  • Polling hardware in Model::tick() and retargeting printf to UART

Prerequisites

  • Done Demo 2 (you can design, generate, build, flash a TouchGFX app).
  • You know your ST-LINK COM port number (from Install & verify).

Steps

🎨 TouchGFX Step 1 — Design the screen

Open your project (or a fresh board-template one) in TouchGFX. Add a Button and a TextArea named statusTxt (enable its wildcard buffer, size ≥ 9 to hold RELEASED). Add an Interaction: trigger Button is clicked → action Call new virtual function named ledButtonClicked. Generate Code, then press F5 in CubeIDE.

⚙️ CubeMX Step 2 — Enable the pins & UART

Open the project's .ioc in standalone STM32CubeMX. Set:

  • PI1 → GPIO_Output (on-board LED LD1)
  • PI11 → GPIO_Input (user button B1; reads high when pressed — external pull-down)
  • USART1 → Mode Asynchronous; confirm pins PA9 (TX) / PB7 (RX) — these reach the ST-LINK virtual COM port — at 115200 baud, 8N1

Click Generate Code, then back in CubeIDE press F5. This is the round-trip: a new capability on the screen needed new hardware, so you dropped into CubeMX.

⌨️ CubeIDE Step 3 — Retarget printf to UART

In Core/Src/main.c, add the include and the low-level putchar inside the USER CODE regions:

/* USER CODE BEGIN Includes */
#include <stdio.h>
/* USER CODE END Includes */
/* USER CODE BEGIN 4 */
int __io_putchar(int ch)
{
    HAL_UART_Transmit(&huart1, (uint8_t *)&ch, 1, HAL_MAX_DELAY);
    return ch;
}
/* USER CODE END 4 */

⌨️ CubeIDE Step 4 — Model: read the button, drive the LED

In TouchGFX/gui/include/gui/model/ModelListener.hpp add a callback:

virtual void notifyButtonStateChanged(bool pressed) {}

In TouchGFX/gui/include/gui/model/Model.hpp, add to the class:

public:
    void tick();
    void toggleLed();
private:
    bool lastButtonState = false;

In TouchGFX/gui/src/model/Model.cpp (the constructor already exists — add the methods and the includes):

#include <gui/model/Model.hpp>
#include <gui/model/ModelListener.hpp>
#include "main.h"   // HAL + GPIOI / GPIO_PIN_1 / GPIO_PIN_11
#include <stdio.h>

void Model::tick()
{
    bool pressed = (HAL_GPIO_ReadPin(GPIOI, GPIO_PIN_11) == GPIO_PIN_SET); // B1 active-high
    if (pressed != lastButtonState) {
        lastButtonState = pressed;
        if (modelListener) modelListener->notifyButtonStateChanged(pressed);
    }
}

void Model::toggleLed()
{
    HAL_GPIO_TogglePin(GPIOI, GPIO_PIN_1); // LD1
    printf("LED toggled\r\n");
}
Why Model::tick() The framework calls it once per frame — the canonical place to poll hardware. It compares the button state and only notifies on a change.

⌨️ CubeIDE Step 5 — Presenter: relay both directions

In Screen1Presenter.hpp add:

virtual void notifyButtonStateChanged(bool pressed);
void toggleLed();

In Screen1Presenter.cpp add:

void Screen1Presenter::notifyButtonStateChanged(bool pressed)
{
    view.setButtonState(pressed);
}

void Screen1Presenter::toggleLed()
{
    model->toggleLed();
}

⌨️ CubeIDE Step 6 — View: button → LED, state → text

In Screen1View.hpp add:

void setButtonState(bool pressed);
virtual void ledButtonClicked();   // the interaction stub

In Screen1View.cpp add:

void Screen1View::ledButtonClicked()
{
    presenter->toggleLed();
}

void Screen1View::setButtonState(bool pressed)
{
    Unicode::snprintf(statusTxtBuffer, STATUSTXT_SIZE, pressed ? "PRESSED" : "RELEASED");
    statusTxt.invalidate();
}

⌨️ CubeIDE Step 7 — Build & flash

Build (hammer), then Run (flash). Open a serial terminal (Tera Term / PuTTY) on your ST-LINK COM port at 115200 8N1.

✅ Checkpoint Tap the on-screen button → LED LD1 toggles and LED toggled prints in the terminal. Press the physical B1 button → the screen shows PRESSED / RELEASED.

What just happened

  • UI → hardware: View ledButtonClicked() → Presenter toggleLed() → Model toggleLed() → HAL_GPIO_TogglePin.
  • Hardware → UI: Model::tick() reads B1 → ModelListener → Presenter notifyButtonStateChanged() → View setButtonState() → invalidate().
  • The round-trip: each new pin meant a trip to CubeMX, a Generate, an F5, then code in gui/. That is the loop you'll repeat for every feature.
