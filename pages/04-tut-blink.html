Tutorial 1 — Blink the on-board LED

Goal: blink the on-board user LED to prove your whole toolchain (configure → build → flash → debug) works.

What you'll learn

  • Creating a project from the board selector in STM32CubeIDE
  • Configuring a GPIO pin as an output in CubeMX
  • Understanding the user-code regions that survive code regeneration
  • Building the project and reading the Console output
  • Flashing the firmware to the board and running it
  • A first taste of the step debugger

Prerequisites

  • Finished the Install & Setup guide (STM32CubeIDE installed and working)
  • STM32F746G-DISCO board connected to your PC via the CN14 USB ST-LINK port

Steps

Step 1 — New project

Open STM32CubeMX and click Access to Board Selector. In the Commercial Part Number field, type STM32F746G-DISCO, select the board that appears in the list, then click Start Project.

Step 2 — Initialize peripherals

CubeMX will ask:

"Initialize all peripherals with their default mode?"
Choose No to keep the project minimal.

Note Choosing Yes also works; CubeMX then defines handy pin macros like LD1_Pin for you automatically.

Step 3 — Configure the LED pin

The .ioc file opens automatically in the Pinout & Configuration view. Find pin PI1 on the chip diagram (you can use the search box at the top of the pinout view). Click PI1 and choose GPIO_Output from the pop-up menu.

PI1 is the microcontroller pin wired to the green user LED labelled LD1 on the board's silkscreen.

Step 4 — Generate code

Save the .ioc file with Ctrl+S. A dialog will ask "Do you want to generate Code?". Click Yes. CubeIDE runs CubeMX in the background and writes the project skeleton, including Core/Src/main.c.

Step 5 — Add the blink code

Open Core/Src/main.c in the Project Explorer. Scroll down to the while(1) main loop and add the two lines shown below. The finished block should look exactly like this:

/* USER CODE BEGIN WHILE */
while (1)
{
  HAL_GPIO_TogglePin(GPIOI, GPIO_PIN_1);   /* LD1 user LED on PI1 */
  HAL_Delay(500);                          /* wait 500 ms */
  /* USER CODE END WHILE */

  /* USER CODE BEGIN 3 */
}
/* USER CODE END 3 */
Watch out Only edit between the /* USER CODE BEGIN */ and /* USER CODE END */ markers — regenerating code erases everything outside them.

Step 6 — Build

Go to Project ▸ Build Project (or click the hammer icon in the toolbar). Watch the Console view at the bottom of the screen. A successful build ends with:

Build Finished. 0 errors, 0 warnings.

If you see errors, double-check that you placed the two new lines inside the correct user-code markers in Step 5.

Step 7 — Flash & run

Click Run ▸ Run (the green play button in the toolbar). The first time you do this, CubeIDE opens a Run Configuration dialog — accept the defaults (ST-LINK, SWD interface) and click OK. The IDE compiles if needed, uploads the binary over the ST-LINK connection, resets the chip, and starts execution.

✅ Checkpoint The green LED LD1 on your STM32F746G-DISCO blinks about once per second (500 ms on, 500 ms off).

What just happened

Here is what the code is doing at runtime:

  • main() is called once at power-up, after the startup file has initialised RAM and the HAL clock. It runs through one-time setup code, then falls into the loop.
  • while(1) creates an infinite loop — the firmware never exits; embedded systems almost never do.
  • HAL_GPIO_TogglePin(GPIOI, GPIO_PIN_1) reads the current output state of pin PI1 and flips it: high becomes low, low becomes high. The LED is wired so that high voltage turns it on.
  • HAL_Delay(500) blocks for 500 milliseconds. Under the hood it counts ticks from the SysTick timer, a hardware counter that fires every 1 ms by default.
  • HAL (Hardware Abstraction Layer) is the C library that ST supplies with every STM32. It wraps the raw hardware registers in readable function calls so you do not have to read the 1 000-page reference manual on day one.

Try debugging

STM32CubeIDE includes a full GDB-based debugger that can pause the CPU and inspect variables live on the hardware.

  1. Click Run ▸ Debug (the bug icon). Accept the debug configuration the same way you accepted the run configuration.
  2. CubeIDE switches to the Debug perspective and halts at the start of main().
  3. Click in the grey gutter to the left of the HAL_Delay(500); line to set a breakpoint (a blue dot appears).
  4. Press F8 (Resume) to let the program run. It will stop each time it hits the breakpoint.
  5. Use F6 (Step Over) to advance one source line at a time and watch the LED pin toggle in the SFR (Special Function Registers) view.
  6. Press F8 again to resume free-running, or Ctrl+F2 to stop the debug session.
Tip You can hover over any variable in the source editor while paused at a breakpoint and the debugger will show its current value in a tooltip.
