Workflow & project structure

For a display board, the productive loop is TouchGFX-first: design the UI, let it scaffold the project, then add peripherals in CubeMX and behaviour in CubeIDE. Here is the whole loop.

The optimal flow

  1. 🎨 TouchGFX Create a project from the STM32F746G-DISCO application template. It generates the .ioc and the CubeIDE project for you.
  2. 🎨 TouchGFX Design screens, widgets, and interactions → Generate Code.
  3. ⌨️ CubeIDE Import the generated project, Build (hammer), Run (flash over ST-LINK).
  4. ⚙️ CubeMX Need a pin or peripheral (GPIO, UART, I²C…)? Open the .ioc in standalone CubeMX, enable it, Generate Code.
  5. ⌨️ CubeIDE Write your behaviour in TouchGFX/gui/ (and in Core USER CODE regions for low-level glue). Build, flash, repeat.
After every Generate CubeIDE does not auto-refresh. Select the project in the Project Explorer and press F5 (Refresh) so new files appear. (True for both TouchGFX and CubeMX generation.)

What the project looks like

A TouchGFX-scaffolded project mixes files owned by three tools. Knowing who owns what tells you where your code goes and what gets overwritten.

Two rules that save hours Never edit TouchGFX/generated/ or Drivers/ — your changes vanish on the next Generate. And after any Generate, press F5 in CubeIDE.
When do I go back to TouchGFX vs CubeMX vs CubeIDE? Layout/widgets/animations → TouchGFX. A new pin, peripheral, or clock change → CubeMX (the .ioc). Logic, hardware reads/writes, talking between screen and hardware → C++ in gui/ (CubeIDE). Demo 3 walks the full loop.
